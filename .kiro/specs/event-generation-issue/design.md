# Event Generation Issue Bugfix Design

## Overview

The Sentinel-Core event processing pipeline is losing approximately 90% of eBPF security events between Tetragon generation and API storage. When 5 attack commands execute in the Kubernetes cluster, Tetragon generates ~10 eBPF events, but only 1 event appears in the _recent_events deque and is returned by the /api/events endpoint. This design formalizes the bug condition, analyzes potential root causes in the event processing pipeline (stream reading, JSON parsing, event transformation, or recording), and outlines a systematic approach to identify and fix the event loss while preserving all existing functionality.

## Glossary

- **Bug_Condition (C)**: The condition where eBPF events generated by Tetragon fail to be recorded in _recent_events - when multiple events are generated but only 1 is stored
- **Property (P)**: The desired behavior where all valid eBPF events from Tetragon are successfully processed through process_line() and recorded via record_event()
- **Preservation**: Existing event processing behaviors that must remain unchanged - metrics tracking, SSE broadcasting, ML triage, Redis publishing
- **process_line()**: The function in `forwarder/main.py` that parses JSON lines, transforms events, runs ML triage, publishes to Redis, and calls record_event()
- **record_event()**: The function in `forwarder/api.py` that stores events in _recent_events deque and updates metrics counters
- **transform_event()**: The function in `forwarder/transformer.py` that converts raw Tetragon JSON to Sentinel schema, returning None for irrelevant event types
- **_recent_events**: A deque(maxlen=100) in `forwarder/api.py` that stores processed events for API queries
- **Tetragon**: The eBPF security agent that generates process_exec, process_kprobe, and other kernel-level security events

## Bug Details

### Bug Condition

The bug manifests when multiple eBPF events are generated by Tetragon in response to attack commands, but the Sentinel pipeline fails to record most of them in the _recent_events deque. The event loss occurs somewhere in the pipeline between kubectl logs streaming and the record_event() call.

**Formal Specification:**
```
FUNCTION isBugCondition(pipeline_state)
  INPUT: pipeline_state containing {
    tetragon_events_generated: int,
    events_in_recent_events: int,
    attack_commands_executed: int
  }
  OUTPUT: boolean
  
  RETURN pipeline_state.attack_commands_executed >= 5
         AND pipeline_state.tetragon_events_generated >= 8
         AND pipeline_state.events_in_recent_events < 3
         AND (pipeline_state.events_in_recent_events / pipeline_state.tetragon_events_generated) < 0.3
END FUNCTION
```

### Examples

- **Example 1**: 5 attack commands execute (curl, cat /etc/shadow, nc, ps aux, nslookup) → Tetragon generates 10 events → Only 1 event in _recent_events → Expected: 10 events in _recent_events
- **Example 2**: curl command executes → Tetragon generates process_exec event → Event is filtered out by transform_event() returning None → Expected: Event should be recorded if it's a relevant security event
- **Example 3**: Multiple events stream from kubectl logs → JSON parsing fails silently for some lines → Only 1 event successfully parsed → Expected: All valid JSON lines should be parsed
- **Edge Case**: 10 events generated but all are test/irrelevant event types → 0 events recorded → Expected: This is correct behavior if events are legitimately irrelevant

## Expected Behavior

### Preservation Requirements

**Unchanged Behaviors:**
- Metrics tracking must continue to increment events_total, events_by_type, severity_breakdown, and triage_breakdown counters
- SSE broadcasting must continue to push events to all active WebSocket/SSE subscribers via asyncio queues
- ML triage must continue to add triage and explanation fields to sentinel_event objects
- Redis publishing must continue to call publisher.publish(sentinel_event) for downstream consumers
- Event transformation must continue to filter out irrelevant event types by returning None

**Scope:**
All event processing behaviors that do NOT involve the bug condition (event loss) should be completely unaffected by this fix. This includes:
- The structure and format of recorded events in _recent_events
- The order of operations in process_line() (parse → transform → triage → publish → record)
- The behavior of the /api/events, /api/metrics, and SSE endpoints
- Error handling for malformed JSON (record_error() calls)

## Hypothesized Root Cause

Based on the bug description and code analysis, the most likely issues are:

1. **Event Type Filtering in transform_event()**: The transformer may be returning None for legitimate security events, causing them to be silently dropped in process_line()
   - The _detect_event_type() function only recognizes 5 event types: process_exec, process_exit, process_kprobe, process_tracepoint, process_loader
   - Attack commands may generate other Tetragon event types that are not in this list
   - When transform_event() returns None, process_line() returns early without recording the event

2. **Stream Buffering Issues**: The kubectl logs stream may be buffering events, and the Python stdin reader may not be flushing properly
   - The subprocess.Popen uses bufsize=1 (line-buffered) but stdin reading may still batch events
   - Events may arrive in bursts but only the first event in each batch is processed

3. **JSON Parsing Failures**: Some event lines may fail JSON parsing silently, incrementing errors_total but not surfacing the issue
   - Malformed JSON lines are caught by the try/except in process_line() and logged as warnings
   - If most events have malformed JSON, only 1 valid event would be recorded

4. **Race Condition in record_event()**: Multiple events may be processed concurrently, causing some to be lost when appending to _recent_events
   - The _lock is used in record_event() to protect _recent_events.appendleft()
   - However, if events are processed in separate threads without proper synchronization, some may be lost

5. **TracingPolicy Filtering**: The sentinel-policy.yaml may be configured to only capture specific event types, causing Tetragon to not generate events for most attack commands
   - This would mean the bug is upstream in Tetragon configuration, not in the Sentinel pipeline

## Correctness Properties

Property 1: Bug Condition - All Generated Events Recorded

_For any_ pipeline execution where Tetragon generates N valid security events (N >= 8) in response to attack commands, the fixed pipeline SHALL record all N events in the _recent_events deque, making them available via the /api/events endpoint.

**Validates: Requirements 2.1, 2.2, 2.3, 2.4**

Property 2: Preservation - Event Processing Behavior

_For any_ event that is successfully recorded (both before and after the fix), the fixed pipeline SHALL produce exactly the same metrics updates, SSE broadcasts, ML triage results, and Redis publications as the original pipeline, preserving all downstream event processing functionality.

**Validates: Requirements 3.1, 3.2, 3.3, 3.4, 3.5**

## Fix Implementation

### Changes Required

Assuming our root cause analysis is correct, the fix will likely involve one or more of these changes:

**File**: `cspm-ebpf/forwarder/transformer.py`

**Function**: `_detect_event_type()` and `transform_event()`

**Specific Changes**:
1. **Expand Event Type Recognition**: Add logging to identify which event types are being dropped
   - Add a logger.debug() statement before returning None to log the raw event keys
   - Run the pipeline and observe which event types are present but not recognized
   - Add missing event types to the known_types list in _detect_event_type()

2. **Add Event Type Statistics**: Track how many events are filtered vs. recorded
   - Add a counter in api.py for filtered_events_total
   - Increment this counter when transform_event() returns None
   - Expose this metric in /api/metrics to diagnose filtering issues

**File**: `cspm-ebpf/forwarder/main.py`

**Function**: `process_line()` and `stream_from_stdin()`

**Specific Changes**:
3. **Enhanced Logging for Dropped Events**: Add detailed logging when events are filtered
   - Log the full raw event JSON when transform_event() returns None
   - This will help identify if legitimate security events are being filtered

4. **Stream Buffering Fix**: Ensure stdin is read without buffering delays
   - Verify that sys.stdin iteration is not batching events
   - Add flush() calls if necessary to ensure real-time processing

5. **JSON Parsing Error Visibility**: Improve error reporting for malformed JSON
   - Log the actual line content when JSON parsing fails (truncated for safety)
   - Track which lines are failing to parse

**File**: `cspm-ebpf/forwarder/api.py`

**Function**: `record_event()`

**Specific Changes**:
6. **Thread Safety Verification**: Ensure no race conditions in event recording
   - Verify that _lock is properly protecting all _recent_events operations
   - Add logging to track concurrent record_event() calls

## Testing Strategy

### Validation Approach

The testing strategy follows a two-phase approach: first, surface counterexamples that demonstrate the bug on unfixed code by observing event loss, then verify the fix captures all events and preserves existing behavior.

### Exploratory Bug Condition Checking

**Goal**: Surface counterexamples that demonstrate the bug BEFORE implementing the fix. Confirm or refute the root cause analysis by identifying where events are being lost.

**Test Plan**: Add instrumentation logging to each stage of the pipeline (stream reading, JSON parsing, event transformation, event recording) and run start_app.sh on UNFIXED code. Count events at each stage to identify the bottleneck.

**Test Cases**:
1. **Stream Reading Test**: Count lines received from kubectl logs stdin (will show ~10 lines on unfixed code)
2. **JSON Parsing Test**: Count successful JSON.loads() calls vs. exceptions (will show if parsing is the bottleneck)
3. **Event Transformation Test**: Count transform_event() calls that return None vs. valid events (will likely show ~9 None returns)
4. **Event Recording Test**: Count record_event() calls (will show only 1 call on unfixed code)

**Expected Counterexamples**:
- Most events return None from transform_event() due to unrecognized event types
- Possible causes: missing event types in known_types list, incorrect event type detection logic, or upstream Tetragon filtering

### Fix Checking

**Goal**: Verify that for all inputs where the bug condition holds (multiple events generated), the fixed pipeline records all events.

**Pseudocode:**
```
FOR ALL pipeline_execution WHERE isBugCondition(pipeline_execution) DO
  attack_commands := execute_5_attacks()
  tetragon_events := count_tetragon_events_generated()
  recorded_events := query_api_events()
  ASSERT len(recorded_events) >= 8
  ASSERT len(recorded_events) / tetragon_events >= 0.8
END FOR
```

### Preservation Checking

**Goal**: Verify that for all events that are successfully recorded, the fixed pipeline produces the same metrics, SSE broadcasts, and Redis publications as the original pipeline.

**Pseudocode:**
```
FOR ALL event WHERE event is successfully recorded DO
  original_metrics := capture_metrics_before_fix(event)
  fixed_metrics := capture_metrics_after_fix(event)
  ASSERT original_metrics.events_total increment == fixed_metrics.events_total increment
  ASSERT original_metrics.events_by_type == fixed_metrics.events_by_type
  ASSERT original_metrics.severity_breakdown == fixed_metrics.severity_breakdown
  ASSERT original_metrics.triage_breakdown == fixed_metrics.triage_breakdown
END FOR
```

**Testing Approach**: Property-based testing is recommended for preservation checking because:
- It generates many test cases automatically across different event types and payloads
- It catches edge cases that manual unit tests might miss (e.g., unusual event structures)
- It provides strong guarantees that metrics and processing behavior is unchanged for all recorded events

**Test Plan**: Observe metrics updates on UNFIXED code for the 1 event that is successfully recorded, then write property-based tests capturing that behavior and verify it holds for all events after the fix.

**Test Cases**:
1. **Metrics Preservation**: Verify events_total, events_by_type, severity_breakdown, triage_breakdown are updated identically
2. **SSE Broadcasting Preservation**: Verify events are pushed to _sse_queues with the same structure and timing
3. **ML Triage Preservation**: Verify triage and explanation fields are added with the same logic
4. **Redis Publishing Preservation**: Verify publisher.publish() is called with the same event structure

### Unit Tests

- Test transform_event() with sample Tetragon JSON for each event type (process_exec, process_kprobe, etc.)
- Test _detect_event_type() with various event structures to ensure all types are recognized
- Test process_line() with valid and malformed JSON to verify error handling
- Test record_event() with concurrent calls to verify thread safety

### Property-Based Tests

- Generate random Tetragon event payloads with different event types and verify all are processed
- Generate random attack sequences and verify event count matches expected output
- Test that metrics updates are consistent across many event types and payloads

### Integration Tests

- Run start_app.sh and verify /api/events returns ~10 events after 5 attack commands
- Run start_app.sh and verify /api/metrics shows events_total >= 8
- Test that SSE subscribers receive all events in real-time
- Test that Redis receives all published events
