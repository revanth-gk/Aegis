# Bugfix Requirements Document

## Introduction

The Sentinel-Core event processing pipeline is not recording all generated eBPF security events. When the application runs via `./start_app.sh` and executes 5 attack commands in the Kubernetes cluster, approximately 10 eBPF events should be generated and captured by Tetragon. However, when querying the API endpoint at http://localhost:8081/api/events, only 1 event is returned instead of the expected ~10 events.

This significantly impacts the security monitoring capability as most attack events are not being made available through the API for analysis and display in the dashboard.

## Bug Analysis

### Current Behavior (Defect)

1.1 WHEN the start_app.sh script executes 5 attack commands (curl, cat /etc/shadow, nc, ps aux, nslookup) in the attacker pod THEN approximately 10 eBPF events are generated by Tetragon

1.2 WHEN these eBPF events flow through the Sentinel pipeline and are processed by process_line() THEN only 1 event gets recorded in the _recent_events deque

1.3 WHEN the /api/events endpoint is called at http://localhost:8081/api/events THEN the system returns only 1 event in the JSON response instead of ~10 events

1.4 WHEN multiple events are generated from the attack sequence THEN most events are lost and not available for API queries or dashboard display

### Expected Behavior (Correct)

2.1 WHEN the start_app.sh script executes 5 attack commands in the attacker pod THEN the system SHALL capture and process all ~10 eBPF events generated by Tetragon

2.2 WHEN eBPF events flow through the Sentinel pipeline and are processed by process_line() THEN the system SHALL record all processed events in the _recent_events deque via record_event()

2.3 WHEN the /api/events endpoint is called at http://localhost:8081/api/events THEN the system SHALL return all ~10 captured events in the JSON response

2.4 WHEN events are recorded THEN the system SHALL maintain them in the _recent_events deque (maxlen=100) so they are available for subsequent API queries

### Unchanged Behavior (Regression Prevention)

3.1 WHEN events are recorded via record_event() THEN the system SHALL CONTINUE TO update metrics counters (events_total, events_by_type, severity_breakdown, triage_breakdown)

3.2 WHEN events are pushed to SSE subscribers THEN the system SHALL CONTINUE TO broadcast events to all active WebSocket/SSE connections for real-time dashboard updates

3.3 WHEN the ML triage processes events THEN the system SHALL CONTINUE TO add triage and explanation fields to sentinel_event objects

3.4 WHEN events are published to Redis THEN the system SHALL CONTINUE TO call publisher.publish(sentinel_event) for downstream consumers

3.5 WHEN the /api/metrics endpoint is called THEN the system SHALL CONTINUE TO return accurate aggregate statistics based on all recorded events
